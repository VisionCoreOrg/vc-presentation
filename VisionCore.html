01

O monitoramento manual falhou em escalar.

CONTEXTO · DIAGNÓSTICO

Pátios comerciais — shoppings, hospitais, centros empresariais — operam hoje com infraestrutura analógica ou semi-digital. A consequência é fila na entrada, custo operacional alto e visibilidade zero do que acontece dentro do pátio em tempo real.

EX—A

Acesso por ticket de papel

Cancelas com leitor de cupom geram filas de 30 a 90 segundos por veículo, custo recorrente de bobina e descarte irregular dentro do pátio.

FRICÇÃO
EX—B

CFTV passivo, reativo

As câmeras já estão instaladas — mas apenas gravam. Não geram alertas, não cruzam dados e dependem de operador humano consultando timestamps após o incidente.

CAPEX OCIOSO
EX—C

Sensores físicos por vaga

Soluções tradicionais exigem indutor magnético ou ultrassônico em cada vaga. Custo de instalação alto, manutenção mecânica constante e baixa tolerância a falhas.

CUSTO · MANUTENÇÃO
02

Hipótese de trabalho.

PROPOSTA · TESE
DECLARAÇÃO ↦ H-01

É possível substituir todo o hardware proprietário de um pátio por uma camada de software que executa visão computacional sobre as câmeras IP/RTSP já instaladas, mantendo latência sub-segundo e operando em hardware de borda comum.

A · 01

A câmera é um sensor — só precisa de inteligência.

Frames RTSP são entrada para modelos de detecção espacial (YOLOv8) e leitura de caracteres (EasyOCR). Não precisamos comprar nada físico.

A · 02

IA pesada não pode bloquear o caminho crítico.

Inferência roda em worker isolado, consumindo de uma fila assíncrona (Redis). A API transacional nunca para para esperar o modelo.

A · 03

Mídia e dado vivem em planos separados.

Imagens vão para MinIO (S3), registros relacionais para MySQL. O banco não engorda com binários, o storage não negocia integridade.

A · 04

Tudo containerizado, tudo reproduzível.

Sete contêineres orquestrados via docker-compose. Um único ponto exposto: nginx_gateway. Subida da stack inteira em < 90s.

03

Arquitetura distribuída orientada a eventos.

DIAGRAMA · TOPOLOGIA

Sete serviços, três planos de responsabilidade. O plano de borda captura, o plano de inferência entende, o plano transacional persiste. Cada um falha isoladamente — e o conjunto se recupera por design.

PL—01

Perímetro · Nginx Gateway

Único ponto exposto à internet. Faz proxy reverso, aplica rate-limit na rota /api/ e roteia /storage/ direto ao MinIO sem passar pela API.

EDGE
PL—02

Mensageria · Redis Broker

Fila FIFO em memória. camera_mock empurra com LPUSH, worker consome com BLPOP de forma bloqueante mas eficiente. Desacopla 100% IA de API.

EVENT
PL—03

Inferência · Worker Portaria

Roda YOLOv8 em ONNX Runtime para localizar a placa e EasyOCR para extrair caracteres. Pesos exportados para reduzir RAM em ~40%.

AI
PL—04

Mídia · MinIO Object Storage

Frames brutos e recortes de placa em buckets S3-compatíveis. URLs assinadas servidas direto pelo Nginx, sem onerar a API Python.

BLOB
04

Pipeline LPR — execução em tempo real.

SIMULAÇÃO · 04 ETAPAS

Acompanhe abaixo o ciclo completo do reconhecimento, do disparo da câmera à abertura da cancela. O simulador à direita responde ao seu scroll — cada etapa avança um estágio do pipeline.

↦ 04.1 · CAPTURA

A câmera observa, registra e enfileira.

O contêiner vc_camera_mock detecta o veículo, captura o frame em alta resolução e dispara dois eventos em paralelo — sem esperar resposta de ninguém.

PROTOCOLO · NÃO-BLOQUEANTE
[CAM]→PUT plate-bucket/frame_0234.jpg em MinIO
[CAM]→LPUSH camera:portaria:queue em Redis
Continue rolando · próximo: inferência
↦ 04.2 · INFERÊNCIA

O worker acorda, vê e lê.

O vc_worker_portaria consome a fila com BLPOP bloqueante. Baixa o frame, roda YOLOv8 para localizar a placa, recorta, e passa o crop pro EasyOCR.

PIPELINE · ONNX RUNTIME
[WRK]←BLPOP camera:portaria:queue
[WRK]→GET frame de MinIO & YOLOv8 inference
[WRK]→EasyOCR → "BRA3E99" · conf 0.985
Continue rolando · próximo: persistência
↦ 04.3 · PERSISTÊNCIA

A API valida, grava, autoriza.

Worker emite um POST /api/registros. A API consulta o cadastro relacional no MySQL, decide acesso liberado/negado e grava a linha de histórico com FK para a foto no S3.

TRANSAÇÃO · ACID
[WRK]→POST /api/registros  {placa, conf, url}
[API]→SELECT veículo WHERE placa = 'BRA3E99'
[API]→INSERT registros (camera, placa, status='LIBERADO')
Continue rolando · próximo: feedback
↦ 04.4 · FEEDBACK

O painel atualiza, a cancela abre.

O dashboard React, com polling de 5s via React Query, recebe o novo registro. A cancela física recebe o sinal e o veículo cruza o portal. Fim do ciclo — em < 1 segundo.

05

Stack técnico e decisões arquiteturais.

TECNOLOGIAS · 06 ITENS

Cada escolha abaixo foi feita contra uma alternativa óbvia. Clique para ver a justificativa técnica e o snippet de implementação real.

FastAPI

PYTHON 3.12 · ASGI

Core transacional. Rotas async, dependency injection, OpenAPI auto-gerado.

YOLOv8 + ONNX

VISÃO COMPUTACIONAL

Detecção espacial otimizada via ONNX Runtime. ~40% menos RAM que PyTorch.

Redis

BROKER · IN-MEMORY

Fila FIFO assíncrona. LPUSH/BLPOP para desacoplar inferência da API.

MinIO

OBJECT STORAGE · S3

Self-hosted, API S3 nativa. Frames servidos direto pelo Nginx sem onerar a API.

MySQL 8

RDBMS · ACID

Cadastros relacionais, FK para mídia. SQLAlchemy + pool_pre_ping.

Nginx

REVERSE PROXY · TLS

Gateway único exposto. Rate-limit por zona, roteamento por path.

EVOLUÇÃO · PÓS-DEFESA

FASE 02 · EM ANDAMENTO

Worker de Pátio · vagas livres por IoU
